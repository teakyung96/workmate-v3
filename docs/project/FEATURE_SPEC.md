# 01. 기능 명세서 (Feature Spec)

- **작성일**: 2026-07-14
- **상태**: 사용자 검토 대기
- **상위 문서**: [workmate-design.md](workmate-design.md) — 전체 개요·아키텍처
- **관련 문서**: [02 화면 설계서](02_SCREEN_DESIGN.md) · [03 디자인 가이드](03_DESIGN_GUIDE.md) · [04 API·DB 설계서](04_API_DB_SPEC.md) · [05 공통 요구사항](05_COMMON_REQUIREMENTS.md)

이 문서는 Workmate 5개 기능 영역의 **상세 요구사항**을 정의한다.
각 영역은 `시나리오 → 상세 요구사항 → 예외 케이스 → 완료 조건` 순으로 기술하며, 개발 단계(마일스톤)는 `인증+채팅(1) → 영수증(2) → RAG(3) → Tool Calling(4) → 마무리(5)` 순서를 따른다.

## F1. 회원/인증 (마일스톤 1)

### F1.1 사용자 시나리오

- 신규 사용자가 이메일·비밀번호로 가입하고, 로그인하여 서비스를 사용한다.
- 비밀번호를 5회 연속 틀리면 계정이 1시간 잠기고, 잠금 사유가 화면에 안내된다.
- 같은 계정으로 다른 브라우저에서 로그인하면 기존 세션은 끊긴다(중복 로그인 방지).

### F1.2 상세 요구사항

| ID     | 요구사항                                                                                                                       | 비고                                                                                              |
| ------ | ------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------- |
| F1-01  | 회원가입: 이메일(로그인 ID)·비밀번호·이름·전화번호 입력                                                                        | 이메일 중복 검사                                                                                  |
| F1-01a | **이메일 정규화는 WAS 진입 시점에 서버가 수행**: `trim` + 소문자 변환 후 그 값으로 중복 검사·암호화·저장·로그인 대조 전부 처리 | 클라이언트 검증은 UX 보조일 뿐 방어선 아님. 암호화 저장(UK가 대소문자 중복을 못 잡음) 특성상 필수 |
| F1-02  | 비밀번호는 `BCryptPasswordEncoder` 단방향 암호화 저장                                                                          | 평문·복호화 가능 방식 금지                                                                        |
| F1-03  | 이메일·전화번호는 AES-256 투명 암호화 저장 (v1 이식)                                                                           | JPA `AttributeConverter`                                                                          |
| F1-04  | 비밀번호 정책: 8자 이상, 영문+숫자+특수문자 조합                                                                               | 서버 측 검증 필수                                                                                 |
| F1-05  | 로그인: 세션 기반 인증 (Spring Security, WEB 레이어 담당)                                                                      | 토큰 아님 — WEB이 세션 보유, WAS는 stateless                                                      |
| F1-06  | 로그인 5회 연속 실패 시 계정 잠금 1시간                                                                                        | 실패 횟수·잠금 시각 DB 기록                                                                       |
| F1-07  | 잠금 상태 로그인 시도 → 남은 잠금 시간 안내                                                                                    | 비밀번호 정오 여부는 노출하지 않음                                                                |
| F1-08  | 중복 로그인 방지: 새 로그인 시 기존 세션 만료                                                                                  | Spring Session 세션 레지스트리                                                                    |
| F1-09  | 로그아웃: 세션 무효화 후 로그인 화면 이동                                                                                      |                                                                                                   |
| F1-10  | 미인증 상태로 보호 화면 접근 시 로그인 화면으로 리다이렉트                                                                     | 로그인 후 원래 목적지 복귀                                                                        |

### F1.3 예외 케이스

| 상황                  | 처리                                                                      |
| --------------------- | ------------------------------------------------------------------------- |
| 가입 시 이메일 중복   | "이미 사용 중인 이메일입니다" — 폼 에러 표시, 다른 입력값 유지            |
| 로그인 실패 (잠금 전) | "이메일 또는 비밀번호가 올바르지 않습니다" — 어느 쪽이 틀렸는지 노출 금지 |
| 계정 잠금 중 로그인   | "계정이 잠겼습니다. NN분 후 다시 시도해주세요"                            |
| 세션 만료 후 요청     | 로그인 화면 리다이렉트 + "세션이 만료되었습니다" 안내                     |

### F1.4 완료 조건

> 2026-07-15 수동 E2E 검증 결과 반영 (Task 16). 미충족 2건은 아래 기술 부채 ③·④ 로 등록.

- [x] 가입 → 로그인 → (인증 화면) 진입 → 로그아웃 전체 흐름 동작 — 검증 완료 (현재 로그인 후 랜딩은 영수증 홈; 채팅 화면은 F2 미구현)
- [ ] 비밀번호 5회 실패 → 잠금 → 1시간 후 자동 해제 확인 — ⚠️ **미충족**: 6회 시도 후에도 `login_fail_count=0`, 잠금 미발생 (기술 부채 ③)
- [ ] 브라우저 2개 동시 로그인 시 먼저 로그인한 세션이 끊김 — ⚠️ **미충족**: 두 세션이 동시에 유효, 기존 세션 미만료 (기술 부채 ④)
- [x] `User@Example.com`으로 가입 요청(클라이언트 검증 우회) → 소문자로 저장되고, `user@example.com` 재가입 시 중복 거부·로그인은 대소문자 무관 성공 — 검증 완료
- [x] DB에서 이메일·전화번호가 암호문으로, 비밀번호가 BCrypt 해시로 저장됨 — 검증 완료 (email AES 암호문·password `$2a$...`)

## F2. AI 채팅 + SSE 스트리밍 (마일스톤 1)

### F2.1 사용자 시나리오

- 사용자가 사이드바의 **[+ 새 채팅]** 버튼으로 새 대화를 시작해 질문하면, AI 응답이 **글자 단위로 실시간** 표시된다.
- 사이드바에서 과거 채팅방을 선택해 이력을 다시 보고 이어서 대화한다.
- 필요 없는 채팅방은 삭제한다(목록에서 사라짐).

### F2.2 상세 요구사항

| ID    | 요구사항                                                                                               | 비고                                                     |
| ----- | ------------------------------------------------------------------------------------------------------ | -------------------------------------------------------- |
| F2-01 | **[+ 새 채팅] 버튼**: 사이드바 상단 고정. 클릭 시 빈 채팅 화면으로 전환 (진행 중이던 방은 목록에 유지) | 로그인 직후 기본 화면도 이 빈 채팅 상태                  |
| F2-02 | 채팅방 생성: 빈 채팅 화면에서 **첫 메시지를 전송하는 순간** DB에 방 생성, 제목은 첫 질문 앞 30자       | 버튼만 누르고 질문 안 하면 방이 생기지 않음 — 빈 방 방지 |
| F2-03 | 채팅방 목록: 사이드바에 최신순, 본인 것만                                                              | 사이드바 통합형 레이아웃                                 |
| F2-04 | 채팅방 삭제: 논리 삭제(`use_yn`)                                                                       | 이력 데이터 보존                                         |
| F2-05 | 메시지 전송 → Spring AI `ChatClient`(Gemini 2.5 Flash) 호출                                            | `spring-ai-starter-model-google-genai`                   |
| F2-06 | 스트리밍: WAS `Flux` → WEB 무버퍼 relay → 브라우저 SSE                                                 | 두 홉 실시간 통과, WEB 버퍼링 금지                       |
| F2-07 | 응답은 마크다운 렌더링 (코드블록·표·목록)                                                              | 클라이언트 렌더러 사용                                   |
| F2-08 | 대화 이력 저장: user/assistant 메시지 모두 DB 기록                                                     | `chat_message.role`                                      |
| F2-09 | 응답 생성에 사용한 모델명 기록 (`model_name`)                                                          | 마일스톤 4 멀티모델 대비                                 |
| F2-10 | 이전 대화 맥락 포함 호출: 해당 방의 최근 N개 메시지를 프롬프트에 포함                                  | N은 토큰 한도 고려해 설정값으로                          |
| F2-11 | 사용자별 분당 AI 요청 횟수 제한                                                                        | 쿼터 보호·악용 방지                                      |

### F2.3 예외 케이스

| 상황                            | 처리                                                                            |
| ------------------------------- | ------------------------------------------------------------------------------- |
| Gemini API 실패 (타임아웃·쿼터) | 재시도 1회 → 실패 시 "잠시 후 다시 시도해주세요" + 로그에 원인 구분 기록        |
| 스트리밍 중간 끊김              | SSE error 이벤트 → **수신분 보존** + "응답이 중단되었습니다" 표시 + 재시도 버튼 |
| 요청 제한 초과                  | "요청이 많습니다. 잠시 후 이용해주세요" (HTTP 429)                              |
| 빈 메시지 전송                  | 전송 버튼 비활성화 (클라이언트) + 서버 측 검증                                  |

### F2.4 완료 조건

- [ ] [+ 새 채팅] 클릭 → 빈 화면 전환, 첫 메시지 전송 시에만 방 생성 (전송 없이 이탈하면 방 미생성)
- [ ] 로그인 → 질문 → 글자 단위 스트리밍 응답 → 새로고침 후 이력 유지
- [ ] 채팅방 2개 이상에서 각 방의 맥락이 섞이지 않음
- [ ] 스트리밍 중 WAS 강제 종료 시 수신분 보존 + 재시도 버튼 노출

## F3. 영수증 자동 인식 (마일스톤 2) — 대표 기능 ⭐

### F3.1 사용자 시나리오

- 사용자가 영수증 사진을 업로드하고 **[분석하기]**를 누르면 AI(Vision)가 결제 건들을 추출하고, **그 시점에 이력이 자동 저장된다** — 별도 [저장하기] 버튼 없음.
- "롯데법인카드" 결제 건이 1건이면 자동 선택·확정되고, 사용자는 결과 화면에서 금액·사업자번호·결제일을 복사해 회사 ERP에 붙여넣는다.
- 추출값이 틀리면 결과 화면에서 바로 수정한다 — 수정은 이미 저장된 이력 건의 UPDATE.
- 이력 화면에서 과거 건을 다시 보고, 월별 CSV로 내려받는다.

### F3.2 상세 요구사항

**흐름**: 이미지 업로드 → [분석하기] → Vision 추출 + 규칙 검증 → **이력 자동 저장** → 결과 표시(수정 가능한 폼) → 수정 시 UPDATE

| ID     | 요구사항                                                                                                                                     | 비고                                                  |
| ------ | -------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------- |
| F3-01  | 이미지 업로드: jpg/png/webp, 최대 10MB                                                                                                       | 서버 측 확장자·크기 검증                              |
| F3-01a | 업로드한 이미지는 선택 즉시 **이미지 미리보기**로 표시 — 파일명 텍스트만 표시 금지                                                           | 분석 요청 전 잘못 올린 파일을 눈으로 거를 수 있게     |
| F3-01b | 확인 폼에서 **원본 이미지와 추출값 폼을 나란히 표시** — 이미지를 보면서 값 대조·수정                                                         | 확대(클릭 시 원본 크기) 지원, 상세 레이아웃은 02 문서 |
| F3-02  | Vision 추출 스키마: **영수증 여부(`isReceipt`)** + 결제 건 배열 `[{카드사명, 금액, 결제일}, ...]`                                            | 영수증에 복수 결제 수단 존재 가능                     |
| F3-02a | 영수증 여부 판별: `isReceipt=false`(영수증 아닌 이미지)이면 결제 건 추출 없이 안내 표시                                                      | 별도 분류 호출 없이 같은 Vision 호출 한 번으로 판별   |
| F3-03  | ERP 입력 3필드 확정: **금액, 사업자번호, 결제일**                                                                                            |                                                       |
| F3-04  | 카드 선택 규칙: "롯데법인카드" 포함 건이 **정확히 1건**이면 자동 선택(`select_type=AUTO`)                                                    |                                                       |
| F3-05  | 해당 건이 0건 또는 2건 이상이면 자동 추측 금지 → 사용자 선택 폼(`select_type=MANUAL`)                                                        | AI가 임의 판단하지 않음                               |
| F3-06  | 사업자번호 체크섬 검증 (한국 사업자등록번호 알고리즘)                                                                                        | 순수 함수로 구현 — TDD 대상                           |
| F3-07  | 체크섬 실패 시 저장은 허용하되 경고 표시 (`biz_no_valid=false`)                                                                              | 오인식일 수 있으므로 차단하지 않음                    |
| F3-08  | **[분석하기] = 이력 저장**: 분석 완료 시점에 이력 자동 저장(`raw_json` 포함), 별도 저장 버튼 없음. 단 `isReceipt=false`는 이력에 남기지 않음 | 분석만 하고 떠난 건도 남음 — 디버깅 근거 확보         |
| F3-08a | 결과 화면의 추출값은 **수정 가능한 폼** — 수정하면 저장된 이력 건을 UPDATE                                                                   | AI 제안 → 규칙 검증 → 사람 확인 3단계는 유지          |
| F3-08b | MANUAL 케이스(롯데카드 0건/복수 건): 분석 시점엔 결제 필드 **미확정 상태**로 저장 → 사용자가 결제 건을 선택하는 순간 UPDATE로 확정           | 이력 목록에서 미확정 건은 "확인 필요" 뱃지 표시       |
| F3-09  | AI 추출 원본 JSON 전체를 `raw_json`(jsonb) 보관                                                                                              | 규칙 변경 시 재처리·오인식 디버깅 근거                |
| F3-10  | 이력 목록: 본인 것만, 최신순, 결제일·금액·사업자번호·검증 상태 표시. 건 선택 시 **원본 이미지 미리보기** 포함 상세 확인                      | 오인식 여부를 나중에도 원본으로 검증 가능             |
| F3-11  | 건별 필드 복사 버튼 (금액/사업자번호/결제일 각각 클립보드 복사)                                                                              | ERP 수기 입력 보조                                    |
| F3-12  | 월별 CSV 다운로드 (해당 월의 본인 이력, **확정 건만** — 미확정 건은 제외)                                                                    | UTF-8 BOM — 엑셀 한글 깨짐 방지                       |
| F3-13  | 화면 구성: 한 화면 + **[분석]/[이력]** 탭 전환 (저장 단계가 없으므로 "새 등록" 아닌 "분석")                                                  | 화면 설계서 참조                                      |
| F3-14  | 영수증 이미지는 본인만 접근 가능 — 직접 URL 차단, 인증 API로만 다운로드                                                                      | 보안 요구                                             |

### F3.3 예외 케이스

| 상황                                         | 처리                                                                                                                               |
| -------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| **영수증이 아닌 이미지** (`isReceipt=false`) | "영수증으로 인식되지 않았습니다" 안내창 + [다시 업로드] / [그래도 수동 입력] 선택지 — 자동 저장·자동 폴백하지 않음                 |
| OCR 추출/파싱 실패 (API 오류·JSON 깨짐)      | 예외 대신 **빈 수동 입력 폼 폴백** — 이미지는 보관. 수동 입력 경로만은 [등록] 액션으로 이력 생성 (분석 결과가 없어 자동 저장 불가) |
| 영수증은 맞는데 추출 결과 0건                | 수동 입력 폼 + "결제 정보를 인식하지 못했습니다" 안내 (이력 생성은 위와 동일하게 [등록] 액션)                                      |

## F7. 공통코드 (마일스톤 4 — 첫 사용처와 함께 구현)

### F7.1 배경·목적

운영 중 항목이 늘어나는 **표시용 목록**(드롭다운 옵션 등)을 값 종류마다 테이블을 만들지 않고, `그룹 → 코드` 2테이블 + 단일 조회 API로 공통 관리한다. 첫 고객은 `AI_MODEL`(답변 모델 선택 드롭다운)이며, 이후 표시용 목록이 생길 때마다 그룹만 추가한다.

**적용 기준 (중요)** — 모든 상수를 공통코드로 보내지 않는다:

| 구분                             | 저장 위치       | 이유                                                                                          | 예                                                 |
| -------------------------------- | --------------- | --------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| 운영 중 늘어나는 **표시용 목록** | 공통코드 테이블 | 추가 = INSERT, 재배포 불필요                                                                  | `AI_MODEL`                                         |
| **비즈니스 로직이 분기**하는 값  | Java enum 유지  | 코드를 추가해도 분기 로직은 어차피 코드 수정 필요 — 테이블에 두면 로직과 데이터가 분리돼 위험 | `select_type`(AUTO/MANUAL), `role`(user/assistant) |

### F7.2 상세 요구사항

| ID    | 요구사항                                                                                           | 비고                                                 |
| ----- | -------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| F7-01 | 테이블 2개: `common_code_group`(그룹) / `common_code`(코드, 복합 PK: group_code+code)              | 스키마는 04 문서에서 확정                            |
| F7-02 | 코드는 `code`(값)·`code_name`(표시명)·`sort_order`·`use_yn`만 가진다 — 부가 속성 컬럼 없음         | YAGNI — 필요해지는 시점에 추가                       |
| F7-03 | `AI_MODEL` 그룹의 `code`에는 **실제 모델 ID를 그대로** 넣는다 (예: `gemini-2.5-flash`)             | 드롭다운 value가 곧 호출 파라미터 — 매핑 계층 불필요 |

| F7-06 | 코드 등록·수정은 초기에는 SQL(`db/init`)로 관리, 관리자 화면은 범위 외                             | YAGNI — 필요 시 F6 관리자에 추가                     |
