# ADR-0002. 프론트 구조(기능별 모듈 + 공통 모듈) & UI(shadcn-vue + Tailwind)

- **상태**: 채택 (2026-07-21)
- **맥락 문서**: [프론트 구조 가이드](../../development/02_FRONTEND_STRUCTURE_GUIDE.md)

## 맥락 (Context)

Vue3 SPA는 처음이라, 백엔드의 `Controller>Service>DAO`처럼 **정규화된 디렉토리·계층 표준**이 필요하다.
또한 기존 디자인 가이드를 최신 디자인 시스템으로 끌어올리고 싶다.

## 결정 1 — 기능별 모듈 + 공통 모듈 구조

- `src/modules/{auth,chat,receipt,guide,admin}/` 로 **기능별 콜로케이션**(관련 view·composable·store·api·types를 한 폴더에).
- 안내창·페이징·버튼 등 **공통 부품은 `src/common/`** 으로 분리.
- 모듈 내부에서도 계층(`view→composable/store→api`) 유지, 모듈 간 내부 import 금지.

### 타입별 대신 기능별을 택한 이유

- **유지보수·소스 분석 용이**: "채팅 관련은 `modules/chat/` 한 폴더에" → 새 개발자 파악이 빠르다.
- **모듈 경계 습관 확립**: 소규모 프로젝트에서도 확장 가능한 구조를 규율 있게 연습한다(의도된 학습 목표).
- 타입별→기능별 전환 비용은 파일 이동 수준이라, 처음부터 기능별로 시작해도 부담이 적다.

> 규율만 지키면 기능별은 "해로운 과설계"가 아니라 콜로케이션이라는 좋은 습관이다.

## 결정 2 — shadcn-vue + Tailwind v4

- **shadcn-vue** (Reka UI 기반, 구 radix-vue) 채택. 컴포넌트 소스를 복사해 **내가 소유** → `common/components/ui/`에 배치.
- **Tailwind CSS v4** (`@tailwindcss/vite`).
- 기존 디자인 가이드의 **컬러·타이포를 shadcn-vue 테마 토큰으로 매핑** (디자인 가이드를 버리지 않고 테마로 재활용).

### 이유

- 프론트는 주인공이 아니므로 **적은 노력으로 높은 완성도**가 필요 → shadcn-vue가 적합.
- 컴포넌트 소유 방식이 **공통 모듈 개념과 시너지**.
- 접근성(a11y) 내장으로 포트폴리오 품질 향상.

## 고려한 대안

| 대안                              | 기각 이유                                                             |
| --------------------------------- | --------------------------------------------------------------------- |
| 타입별 구조(`views/`·`api/` 모음) | 현 규모엔 무난하나, 모듈 경계 학습·유지보수 콜로케이션 이점을 못 살림 |
| 순수 CSS 유지(v2 방식)            | 완성도 대비 시간 소모 큼. 프론트에 시간 과投입 → AI(주인공) 시간 잠식 |
| React용 shadcn/ui                 | Vue 미지원. Vue3엔 shadcn-vue 사용 필수                               |

## 결과 (Consequences)

- ➕ 유지보수·확장·소스분석 용이, 모던 UI 빠르게 확보, 디자인 가이드 재활용.
- ➖ Tailwind가 딸려와 v2의 순수 CSS 관례를 바꿔야 함(CLAUDE.md·디자인가이드 갱신 완료).
- ➖ shadcn-vue는 커뮤니티 관리(React 대비 살짝 뒤따름) — 안정적이므로 수용.
- 확장 시점: 기능이 크게 늘면 모듈 세분화. 현재 5개 화면 규모엔 현 구조로 충분.
